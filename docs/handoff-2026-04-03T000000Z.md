## Recommended Next Step

Build the next layer above transport:

1. split message-to-world application into a dedicated updater/adapter module
2. deepen the known prim `ObjectUpdate` decode path into full `TextureEntry`, face overrides, and extra params
3. capture and classify any additional live `ObjectUpdate` variants beyond the now-known prim and avatar-style packets
4. continue building durable keyed world entities in `WorldView`
5. connect texture/material references to later asset handling

## Cautions

- the `ObjectAdd` cube spawn path is still experimental and based on inferred shape defaults
- the world model is stronger now, but object state is still partial: prim positions, basic avatar names, and a first default texture UUID are decoded, while full texture-entry semantics, extra params, and richer appearance/state remain incomplete
- avoid collapsing frontend concerns into the session layer; keep the world model UI-agnostic
