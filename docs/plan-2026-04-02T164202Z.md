# Vibestorm Plan

Timestamp: 2026-04-02T16:42:02Z

## Operating Constraints

This project must remain:

- multi-agent compliant
- multi-IDE compliant
- agentic-environment compliant
- easy to hand off between OpenAI, Claude, Gemini, and human contributors

That means we do not hide critical state in one tool's memory, one editor's workspace, or one agent's private assumptions.

## Collaboration Rules

The repository should preserve handoff nodes between contributors.

Required rules:

- Keep major decisions in repo docs, not only in chat.
- Prefer plain files and standard formats over tool-specific metadata.
- Keep protocol artifacts fetchable and reproducible.
- Split work into narrow, restartable stages with explicit outputs.
- Record assumptions, blockers, and next steps at the end of each stage.
- Avoid architecture that requires one long-lived agent session to understand project state.
- Treat generated code as replaceable unless documented and verified.
- Keep interfaces stable enough that different agents can implement adjacent pieces without stepping on each other.

## Repository Structure Plan

Add and maintain these top-level areas as the project grows:

- `docs/`
- `tools/`
- `third_party/`
- `spec/`
- `src/`
- `test/`

Purpose:

- `docs/` holds decisions, plans, protocol notes, and handoff logs.
- `tools/` holds reproducible fetch/build/dev scripts.
- `third_party/` holds pinned external artifacts and mirrors.
- `spec/` holds protocol-focused notes, schemas, and message coverage tracking.
- `src/` holds implementation.
- `test/` holds protocol fixtures, unit tests, and integration harnesses.

## Delivery Strategy

Build the client in layers and declare stable handoff points after each layer.

### Phase 0: Project Grounding

Deliverables:

- project goal document
- protocol research document
- fetch script for canonical protocol artifacts
- timestamped plan

Exit condition:

- a new contributor can understand the project direction and canonical protocol inputs from the repo alone

### Phase 1: Protocol-Core Skeleton

Deliverables:

- chosen implementation language/runtime
- minimal project scaffold
- configuration model
- logging model
- clear module boundaries for login, UDP circuit, capabilities, and event queue

Exit condition:

- the codebase builds and the module layout matches the protocol research

Handoff node:

- another agent should be able to implement any one module without re-deciding the whole architecture

### Phase 2: Login Bootstrap

Deliverables:

- login/bootstrap client
- request/response models for login payloads
- safe config loading for credentials and target grid
- redacted logging

Exit condition:

- the client can obtain bootstrap data needed to reach a simulator

Handoff node:

- login output is serialized in a stable internal model usable by the UDP and capabilities layers

### Phase 3: UDP Circuit

Deliverables:

- packet header encoder/decoder
- sequence tracking
- reliable resend support
- ACK handling
- zerocode support
- message-template loading/parsing

Exit condition:

- the client can encode/decode packets against the official message template and maintain circuit state

Handoff node:

- decoded messages are exposed through a stable API independent of renderer or UI

### Phase 4: Initial Region Connection

Deliverables:

- `UseCircuitCode`
- `CompleteAgentMovement`
- `RegionHandshake` handling
- `RegionHandshakeReply`
- heartbeat and ping support
- periodic `AgentUpdate`

Exit condition:

- the client reaches a stable connected state in a simulator

Handoff node:

- connection state is visible through structured events, not UI-bound callbacks

### Phase 5: Capabilities and Event Queue

Deliverables:

- seed capability resolution
- LLSD client support needed for capability calls
- `EventQueueGet` long-poll loop
- basic event dispatch for simulator/control-plane events

Exit condition:

- the client can stay connected and react to capability-delivered control events

Handoff node:

- capability events feed the same shared event model as UDP messages

### Phase 6: Text/2D Usability

Deliverables:

- text log of region/session state
- chat send/receive
- avatar coarse location view
- region/object summaries
- lightweight 2D map or console dashboard

Exit condition:

- the client is useful without any 3D rendering

Handoff node:

- presentation layer reads from shared world-state models rather than raw packets

### Phase 7: Bounding-Box 3D

Deliverables:

- object update decoding sufficient for transform extraction
- world-object cache
- simple renderer showing objects as boxes

Exit condition:

- the client can visualize a region in rough 3D form

Handoff node:

- rendering consumes normalized world objects rather than protocol structs directly

### Phase 8: Inventory

Deliverables:

- capability-based inventory fetch
- folder/item models
- read-only inventory browser first

Exit condition:

- inventory can be browsed without destabilizing the connection core

## Required Documentation For Every Phase

Every implementation phase should leave behind:

- what changed
- what remains
- known protocol uncertainties
- test status
- exact next recommended step

This keeps the repo usable when switching agents or IDEs.

## Non-Negotiable Engineering Rules

- Do not couple protocol decoding to UI code.
- Do not embed hidden magic values without source references.
- Do not treat unofficial reverse-engineering notes as canonical when official artifacts exist.
- Do not leave partially understood protocol behavior undocumented.
- Do not make one-shot scripts the only path to understanding state.
